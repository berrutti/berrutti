Why I Still Use Plain HTML in 2025

September 14, 2025
I write JavaScript code for my employeer. I love JavaScript and its younger brother TypeScript and, in the end, that technology is what allowed me to move 14 thousand kilometers, across the ocean and start a new life with my wife here in Germany. So I have only nice things to say about it.

Unfortunately it is also true that nowadays it seems impossible to do anything on the web without JavaScript. Not only that but websites rely on fast internet connections for everything. Have you ever ride a train and you saw the 3G indication on your mobile phone (or even worse, the dreadful "E" indicator) and thought "ok there goes my browsing"? Why is that the case. Some years ago 3G was a more than capable connection, but now it feels that unless you are on 4G or 5G, the internet is just plain useless.

I create this blog precisely because of that, as my own capsule of minimalism in a world of social media feeds and algorithmic content distribution. This blog should be readable on a very bad internet connection and without JavaScript. Even if you block CSS files, the content itself should be perfectly readable

In a world obsessed with JavaScript frameworks, I've found that plain HTML still delivers the best user experience for content-focused websites. Here's why:

First, performance. A static HTML page loads instantly. No JavaScript bundle to download, no virtual DOM to render, no hydration step. Just pure, immediate content.

Second, accessibility. Semantic HTML works with screen readers out of the box. No ARIA attributes needed, no complex state management to break keyboard navigation.

Third, longevity. HTML from 1995 still renders perfectly today. Can we say the same about React components from 2015?

The web was built for documents. Sometimes, that's exactly what we need.

The Performance Advantage

When I look at modern websites, I'm often shocked by their loading times. A simple blog post that should be a few kilobytes becomes megabytes of JavaScript. Users on slow connections suffer, mobile users drain their batteries, and search engines penalize the site.

Plain HTML loads in milliseconds. There's no "time to interactive" because it's interactive immediately. There's no layout shift because the content is there from the start.

Future-Proofing

Frameworks come and go. Angular gave way to React, which is now competing with Vue, Svelte, and countless others. But HTML? HTML is eternal.

The first website, created by Tim Berners-Lee in 1991, still works perfectly in modern browsers. That's 30+ years of backward compatibility. Can your React app from 2018 run without updates today?

The Web Components Experiment

I initially built this blog with web components, thinking they'd provide the best of both worlds: modularity without frameworks. I was wrong.

The fundamental problem: web components require JavaScript to function at all. This breaks the entire "plain HTML" philosophy. If you need JavaScript anyway, why not use React?

What I Learned

  • Web components aren't progressive enhancement - They're JavaScript dependencies
  • They don't add value over plain HTML - Just duplicate existing structure
  • Maintenance overhead - Two places to update the same content
  • No actual benefits - Same JS dependency as frameworks, less ecosystem support

So I removed them entirely. This blog is now 100% plain HTML and CSS. No JavaScript. No components. No build step. Just content that works everywhere, instantly.

The key insight: Don't add technology unless it solves a real problem. Web components solved a problem I didn't actually have.

Corollary: The Art of Minimal CSS

Plain HTML is only half the story. The styling philosophy matters just as much.

After years of wrestling with CSS frameworks, I've discovered the power of constraint. Less code, fewer bugs, faster loading. My current CSS philosophy:

  • System fonts only - No web fonts to download
  • No animations unless essential - Performance over flashiness
  • Responsive by default - Mobile-first design
  • Dark mode as a bonus - Not a requirement

Why System Fonts Win

Custom fonts are beautiful, but they come with a cost. Each font file is additional bandwidth, and users see a flash of invisible text while fonts load. System fonts are optimized for the user's device and load instantly.

More importantly, system fonts respect user preferences. If someone has configured their system for better readability, custom fonts often override these settings.

The Animation Trap

CSS animations can enhance user experience, but they're often overused. Every animation is JavaScript in disguise - it triggers repaints and consumes CPU cycles.

I use animations only when they provide clear user value: indicating loading states, showing relationships between elements, or providing feedback for user actions.

Responsive by Design

Instead of adding responsive breakpoints after the fact, I design mobile-first. Every element works on small screens by default. Desktop layouts are enhancements, not requirements.

The result? A 2KB stylesheet that works everywhere and loads instantly.
